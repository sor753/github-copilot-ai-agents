# AGENTS.md

エージェント向けのプロジェクト概要と作業ガイドライン。
詳細なコーディング・デザインルールはインストラクションファイルを参照すること。

---

## プロジェクト概要

**アプリ:** デイリージャーナル（気分トラッカー付き）  
**スタック:** Vite + React + TypeScript + Supabase  
**スタイリング:** Plain CSS（CSS カスタムプロパティ）  
**アーキテクチャ:** 機能別構成（`features/`）。サービス・コンポーネント・フックを分離

---

## 開発環境

Node.js はローカルにインストールしない。すべての開発作業は Docker コンテナ上で行う。

```bash
docker compose up --build   # コンテナのビルドと起動
docker compose up           # 起動（ビルド済みの場合）
docker compose down         # コンテナの停止
docker compose exec app sh  # コンテナへのシェル接続
```

---

## コマンド

コンテナ内、または `docker compose exec app` 経由で実行する。

```bash
npm install          # 依存関係のインストール
npm run dev          # 開発サーバー起動
npm run build        # 本番ビルド
npm run lint         # ESLint 実行
npm run format       # Prettier 実行
npm run typecheck    # 型チェック（tsc --noEmit）
npm run test         # Vitest 実行
```

コミット前に必ず `lint`・`typecheck`・`test` がすべてパスすることを確認する。

## ワークフローの期待値

- **既存パターンに従う。** 新しいコンポーネント・フック・API 関数を追加する前に、同じ機能の既存実装を必ず参照する
- **変更は小さく保つ。** 1タスク1責務。依頼されていないリファクタリングを混入させない
- **大規模リファクタリングは行わない。** 構造的な変更が必要な場合はユーザーに確認してから着手する
- **型の整合性を維持する。** 既存の型定義を壊す変更は行わない
- **テストを追加する。** ロジックを変更した場合は対応するテストを更新または追加する

## HITL（Human-in-the-Loop）

不明点は必ずユーザーに確認すること。ワークアラウンド的な実装はせず、正しい実装をすること。

### 確認を求めるタイミング

以下の単位でユーザーに確認を求める。1回の確認につき1つの判断のみ求める（複数の質問を同時に投げない）。

- 実装方針が定まった時点
- ファイルの新規作成・削除を伴う場合（事前に確認）
- 1機能の実装が完了した時点（次の機能に着手する前に確認）

### バグ対応

エラーやバグの対処時は、上辺だけの修正をしない。根本原因を解明し、コードを根本から修正する。

- 症状を隠す修正（`try/catch` で握りつぶす・型アサーションで回避など）は禁止
- 原因が特定できない場合はユーザーに報告し、調査方針を確認してから進める

## 変更してはいけないもの

以下のファイル・ディレクトリはエージェントが変更・削除してはならない。

- `dist/` — ビルド成果物
- `package-lock.json` / `yarn.lock` — ロックファイル（依存関係の追加・削除時を除く）
- `*.generated.ts` — 自動生成ファイル
- `.env` / `.env.*` — 環境変数ファイル
- `supabase/migrations/` — 適用済みマイグレーション

---

## セキュリティ境界

以下は厳守する。違反はいかなる理由があっても許可されない。

- シークレット・API キー・トークンをコードにハードコードしない。必ず環境変数を使用する
- TypeScript の `strict` モードを弱体化させない（`any`・`@ts-ignore`・`tsconfig` の変更は禁止）
- Supabase の認証フロー・RLS ポリシーをバイパスしない
- `console.log` をコミットに含めない

---

## 変更の報告

タスク完了時に以下を簡潔に報告する。

1. **変更したファイル** — 追加・編集・削除の一覧
2. **変更の理由** — 何を解決したか
3. **未解決の事項** — 判断できなかった点・確認が必要な点
4. **実行したコマンド** — lint / typecheck / test の結果
